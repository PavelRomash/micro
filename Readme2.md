# Задача 1: Обеспечить разработку

### Предлагаемое решение

Для обеспечения процесса разработки предлагаю стэк:
GitLab CI/CD + Self-hosted GitLab Runners (Docker/VM) + GitLab Container Registry + HashiCorp Vault

Выбранный стек соответствует требованиям:

облачная система - GitLab SaaS;
система контроля версий Git - Git VCS GitLab;
репозиторий на каждый сервис - проекты: auth, gateway, users;
запуск сборки по событию из системы контроля версий - push/merge request в GitLab автоматически запускает пайплайн из .gitlab-ci.yml;
запуск сборки по кнопке с указанием параметров - Run pipeline + переменные: DEPLOY_ENV=staging, SKIP_TESTS=true;
возможность привязать настройки к каждой сборке - каждый job имеет свои variables, image, tags;
возможность создания шаблонов для различных конфигураций сборок - include: '/templates/.build-template.yml' + extends: .base_build;
возможность безопасного хранения секретных данных (пароли, ключи доступа) - Masked + Protected переменные GitLab CI/CD. При необходимости — интеграция с HashiCorp Vault;
несколько конфигураций для сборки из одного репозитория - разные job/stage + переменные;
кастомные шаги при сборке - shell-скрипты, вызов Terraform, Ansible;
собственные докер-образы для сборки проектов - image: $CI_REGISTRY/team/custom-builder:1.0 + сборка через docker build внутри GitLab CI;
возможность развернуть агентов сборки на собственных серверах - self-hosted Runner (установка на VM/K8s);
возможность параллельного запуска нескольких сборок - несколько раннеров + concurrent в настройках в конфигурации GitLab Runner;
возможность параллельного запуска тестов - parallel: 5 + фреймворк (pytest --splits, jest --shard).

Организация:

каждый микросервис — отдельный репозиторий
пайплайны описываются в .gitlab-ci.yml
запуск:
автоматически (push / merge request)
вручную с параметрами

Ключевые возможности:

шаблоны пайплайнов (include)
CI/CD переменные для настройки сборок
безопасное хранение секретов (masked, protected)
использование кастомных Docker-образов
self-hosted runners для контроля среды
параллельные сборки и тесты (parallel, matrix)

Обоснование:
GitLab объединяет Git, CI/CD и Registry в одной платформе, что снижает сложность интеграции, обеспечивает масштабируемость и полностью покрывает требования к микросервисной разработке.


# Задача 2: Логи

### Предлагаемое решение

Для обеспечения сбора и анализа логов предлагаю стек:
Promtail + Loki + Grafana

Выбранный стек соответствует требованиям:

сбор логов в центральное хранилище со всех хостов - Promtail устанавливается на каждый хост или запускается как sidecar-контейнер в Kubernetes, читает логи из stdout и отправляет в центральный Loki;
минимальные требования к приложениям, сбор логов из stdout - приложения пишут логи в stdout/stderr, никаких дополнительных библиотек или агентов в код внедрять не нужно;
гарантированная доставка логов до центрального хранилища - Promtail поддерживает буферизацию на диске и повторные попытки отправки при временных сбоях Loki;
обеспечение поиска и фильтрации по записям логов - Loki использует язык запросов LogQL, позволяющий фильтровать по меткам (сервис, хост, уровень) и по содержимому сообщения;
обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам - Grafana выступает единой точкой входа, разработчики получают доступ через встроенный RBAC только к своим сервисам;
возможность дать ссылку на сохранённый поиск по записям логов - в Grafana любой поисковый запрос можно сохранить и скопировать прямую ссылку для передачи коллегам.

Организация:

каждый сервис пишет логи в stdout в структурированном формате (JSON) с обязательными полями timestamp, service, level, traceId;
на каждом узле кластера запущен Promtail, который подхватывает логи контейнеров, добавляет метки (namespace, pod, service) и отправляет в Loki;
Loki принимает логи, индексирует только метки, сжимает и хранит данные в MinIO или любом S3-совместимом хранилище;
Grafana подключается к Loki как к источнику данных и предоставляет интерфейс для поиска и анализа.

Ключевые возможности:

централизованный сбор без изменения кода приложений
низкое потребление ресурсов за счёт отсутствия полной индексации содержимого
интеграция с Grafana — разработчики работают в одном интерфейсе с метриками и логами
ссылки на сохранённые поиски

Обоснование:

Loki индексирует только метки, а не всё содержимое логов, что даёт низкое потребление ресурсов при большом количестве сервисов. 
Promtail забирает логи из stdout без изменений в коде приложений и гарантирует доставку за счёт буферизации.
Grafana предоставляет удобный интерфейс для поиска, фильтрации и создания ссылок на сохранённые запросы.
Всё это закрывает требования к сбору, поиску и анализу логов без избыточной сложности.


# Задача 3: Мониторинг

### Предлагаемое решение

Для обеспечения сбора и анализа состояния хостов и сервисов предлагаю стек:
Prometheus + Node Exporter + cAdvisor + Grafana

Выбранный стек соответствует требованиям:

сбор метрик со всех хостов, обслуживающих систему - Prometheus в режиме pull периодически опрашивает экспортёры на каждом хосте;
сбор метрик состояния ресурсов хостов - Node Exporter собирает CPU, RAM, HDD, Network со всей хост-системы;
сбор метрик потребляемых ресурсов для каждого сервиса - cAdvisor собирает CPU, RAM, HDD, Network для каждого контейнера/сервиса;
сбор метрик, специфичных для каждого сервиса - предоставляют свой /metrics в формате Prometheus, добавляя бизнес-метрики (количество заказов, время ответа, ошибки);
пользовательский интерфейс с возможностью делать запросы и агрегировать информацию - Prometheus имеет встроенный интерфейс с языком запросов PromQL для агрегации и группировки;
пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы - Grafana позволяет создавать дашборды с любыми панелями, графиками, таблицами и алертами.

Организация:

на каждом хосте запущены Node Exporter (метрики хоста) и cAdvisor (метрики контейнеров);
каждый сервис предоставляет /metrics с бизнес-метриками;
Prometheus настроен на сбор со всех этих targets через service discovery или статическую конфигурацию;
Grafana подключается к Prometheus как к источнику данных, разработчики и администраторы строят дашборды.

Ключевые возможности:

единая точка сбора метрик
Pull-модель упрощает обнаружение сервисов
встроенный язык запросов PromQL для агрегации
гибкие дашборды в Grafana

Обоснование:

Prometheus стал стандартом для микросервисной архитектуры благодаря pull-модели, простой настройке экспортёров и мощному языку запросов PromQL.
Node Exporter и cAdvisor закрывают сбор метрик хостов и контейнеров без необходимости писать дополнительный код. Бизнес-метрики каждый сервис предоставляет сам через стандартный эндпоинт.
Grafana даёт удобный интерфейс для создания любых дашбордов и панелей, что полностью закрывает требования к визуализации и агрегации.
